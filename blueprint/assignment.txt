Final Project Defense: Complete Backend & Frontend Implementation

🎯 Objective
This is the final deliverable before your project defense. You must deliver a complete, production-grade backend system integrated with a functional frontend demo, fully deployed and accessible online. Every feature, workflow, and business rule from your approved blueprint must now be implemented, tested, documented, and live on the internet.
Scope: Your project must be 100% feature-complete. The backend must expose every endpoint from your blueprint. The frontend must demonstrate all major user workflows by consuming your live backend API. Your application must be deployed and running on a public URL.
Prerequisite: Your pre-defense backend (authentication, business logic, background workers, real email integration) must remain fully functional and will be tested as the foundation for this final submission.

📋 Description
You are delivering the entire full-stack application for your chosen track. The backend must remain strictly through your ORM (SQLModel or Prisma), with all secrets externalized, all endpoints documented, and all business rules enforced. The frontend is a demonstration interface — it does not need pixel-perfect design, but it must prove that your backend works by allowing real users (and the examiner) to interact with every major feature through a browser.
Your entire stack — backend, frontend, PostgreSQL, and Redis — must be containerized with Docker Compose and deployed to https://dashboard.deployrocks.com so that it boots with a single click.
If it can not be deployed on https://dashboard.deployrocks.com then register a free account at render or vercel (or any other cloud provider) and deploy there.

2. Authentication & Authorization — Must Remain Fully Operational
Your auth subsystem must work perfectly on the deployed live URL and protect all business endpoints:

    Registration / Login / Logout / Refresh: Complete JWT lifecycle with refresh token rotation.
    Email Verification on Signup: During registration, your system must send a real verification email to the user's inbox using a 3rd party email service (e.g., SendGrid, Mailgun, AWS SES, or any provider demonstrated in class). The user must click a verification link or enter a code before their account is fully activated. Unverified users must be blocked from protected routes.
    Password Reset via Email: Users must be able to request a password reset link sent to their real email address.
    RBAC Enforcement: Every role defined in your blueprint (admin, customer, merchant, driver, vendor, etc.) must be implemented. Middleware rejects unauthorized roles with 403.
    Rate Limiting: Redis-based token bucket on all auth endpoints and sensitive public endpoints.
    CORS: Properly configured for your deployed domain. No wildcards in production.

3. Complete Business Logic Implementation
Implement every major workflow from your blueprint. The following table lists the minimum complete feature set per track. If your blueprint promised additional features, they must also be present.
All tracks must send real email notifications for at least 3 business events using the same 3rd party email service (e.g., order confirmation, payment receipt, circle payout alert, food pickup reminder, trust score update, assignment flagged for review, etc.).

Saukele - Full event / wedding registry creation. Hierarchical gift obligation logic with recursive family tree queries (ата-ана, жиен-жаран, құда-жекжең tiers). Pool funding with escrow state machine (pending → funded → purchased → delivered) and partial contribution tracking. Multi-currency snapshots (locked exchange rates at transaction time). Logistics orchestration with fragile-item handling flags. Privacy tiers per gift item. Notification system with culturally-aware timing rules. Emails: registry invitation, pool funding progress update, gift delivery confirmation, gentle payment reminder (culturally timed).

4. API Documentation

    Swagger UI / API Docs: Live at /docs (FastAPI) or mounted endpoint (Express) on your deployed URL. Must document every implemented endpoint.
    Contract Compliance: All endpoints must match your submitted openapi.yaml. Any deviations must be documented in CHANGELOG.md with architectural justification.
    Error Handling: Standardized error responses (400, 401, 403, 404, 409, 422, 500) on every endpoint.
    Pagination: Cursor-based pagination applied consistently to all list endpoints.
    Request/Response Examples: Realistic examples, not placeholder "string" values.

5. Background Workers & Async Processing

    Redis-backed Jobs: At least one working background worker (Celery, BullMQ, or equivalent) handling non-realtime tasks: dead stock decay, price updates, webhook retries, email/SMS reminders, ledger cron jobs, abandoned cart recovery, etc.
    Email Queue: Email sending must be handled asynchronously via the background worker queue. The API endpoint must not block waiting for the 3rd party email service to respond.
    Cron Scheduling: Documented schedule for recurring jobs.
    Queue Visibility: Jobs must be observable (completed, failed, retry counts).

6. Testing & Quality Assurance

    Unit Tests: All pure business logic functions must have tests (formulas, algorithms, scoring rubrics, state machines).
    Integration Tests: Database-touching tests for every critical transaction boundary. Prove that money/inventory/state changes are atomic.
    Auth Integration Tests: Prove that protected endpoints reject missing, expired, wrong-role, and unverified tokens.
    Email Tests: Include tests verifying that email jobs are enqueued correctly (you may mock the 3rd party API call in tests, but the job must be proven to fire).
    CI/CD (Optional): You may include .github/workflows/ci.yml if you wish, but it is not required and will not affect grading.

7. Frontend Demo Application
You must build a functional frontend that consumes your backend API and demonstrates all major user workflows. This is for demonstration purposes during your final defense.

    Framework: Your choice (vanilla HTML/JS, React, Vue etc.). The frontend does not need to be aesthetically polished, but it must be functional and navigable.
    Backend Integration: Every major feature from your backend must be accessible through the frontend:
        User registration, login, logout, email verification, password reset
        Role-based views (e.g., admin dashboard vs. customer view)
        Core business workflows (create, read, update, delete, search, checkout, assign, analyze — depending on track)
    API Consumption: Frontend must make real HTTP requests to your backend endpoints. No mocked data.
    Error Handling: Frontend must display backend error messages (e.g., 401 Unauthorized, validation errors) to the user.
    Responsive (Optional but Recommended): Should work on both desktop and mobile browser views.
    Build & Serve: Frontend must be containerized in your docker-compose.yml and served correctly in production.

    Open your live deployed frontend and confirm it loads.

    Trigger or demonstrate a background worker (show Redis queue or logs). (optional, if asked)
    Run your full test suite live (pytest or jest) and show green results. (optional, if asked)
    Show Swagger UI on the deployed URL with all endpoints documented. (optional, if asked)
    Answer architectural questions about your specific implementation under live questioning.
